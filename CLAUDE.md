## AI Agent 编码行为约束

来源: https://github.com/multica-ai/andrej-karpathy-skills/blob/main/CLAUDE.md

**权衡:** 这些准则偏向谨慎而非速度。对于简单任务，可自行判断。

### 1. 思考先行 (Think Before Coding)

**不要假设。不要隐藏困惑。明确权衡。**

实施之前:
- 明确陈述你的假设。如果不确定，主动询问。
- 如果存在多种解释，呈现它们 — 不要默默选择。
- 如果存在更简单的方法，说出来。必要时提出反对意见。
- 如果某事不清楚，停下来。说出困惑之处。提问。

### 2. 简洁至上 (Simplicity First)

**用最少代码解决问题。不写投机性代码。**

- 不实现超出需求的功能。
- 不为单次使用的代码创建抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景做错误处理。
- 如果写了200行但50行就能解决，重写它。

自问: "高级工程师会说这过于复杂吗？" 如果是，简化。

### 3. 精准修改 (Surgical Changes)

**只动必须动的。只清理你自己造成的混乱。**

编辑现有代码时:
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有问题的东西。
- 匹配现有风格，即使你更倾向不同的写法。
- 如果注意到无关的死代码，提出来 — 不要直接删除。

当你的修改造成孤立代码时:
- 删除被你的修改变成无用的 import/变量/函数。
- 不要删除预先存在的死代码，除非被要求。

检验标准: 每个被改动的行都应直接追溯到用户的请求。

### 4. 目标驱动执行 (Goal-Driven Execution)

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标:
- "添加验证" → "先为无效输入编写测试，然后让测试通过"
- "修复 Bug" → "先写一个能复现的测试，然后让它通过"
- "重构 X" → "确保重构前后测试均通过"

## Git 提交规范

每实现一个独立功能点就做一次 git commit，保持提交粒度细、可独立回滚、便于 code review。

## 沟通规范

默认使用中文沟通，文档默认使用中文，术语和专业名词保持英文原样。

## UI 设计规范

在实现任何 UI 界面之前，先使用 `frontend-design` 技能进行设计，确认设计方案后再开始编码实现。
